What is it?

Usage of third-party libraries, commercial and open-source, has increased in applications over time.

Even the most utilised and trusted components can contain vulnerabilities that compromise their security.

Introducing these libraries improperly can lead to the introduction of the security flaws to all applications which use them.

Projects including multiple dependancies can potentially be introducing large numbers of security vulnerabilities!

Impact

The range of potential components mean that the impact of introduced vulnerabilities can vary depending on the application using them.

It was estimated that 26% of all downloads from the 'central' repository contained a vulnerability.

More information on it's prevalence can be found here.

Exploiters

There are plenty of reasons to target 3rd party libraries for security flaws rather than directly targeting the main application.

Components are often open source and therefore the potential exploiter has access to the code to determine the vulnerabilities.

The potential for any flaw to extend across multiple applications also makes it an appealing target, as the discovery and exploitation of any vulnerability grants a large range of potential targets.

Updates and fixes

Usually after security vulnerabilities are discovered it does not take long for developers to update and fix the issue.

However a lot of the time there is no clear mechanic for informing users of any updates for security purposes.

This means it is quite common for applications to be using out of date dependancies with old security vulnerabilities still in place!

This is a worse scenario as the older issues tend to be better known and therefore more people know how to exploit them.

Undiscovered vulnerabilities

In many cases the vulnerabilities found in component libraries are not yet discovered.

As a result many new breaches are not due to new updates to the library but already existed in the code.

Solution!

There are very few ways to prevent utilisation of vulnerable third party software short of never using any third party dependancies at all.

There are however methods to at least reduce the risk in using libraries.

Following the four simple steps outlined below allows at least an extra degree of control and security.

Additionally GDS guidelines on third party libraries can be found here.

Inventory

The first step is to take stock of your current project dependancies and versions.

This involves looking up known faults and vulnerabilities for the version used as well as how out of date it is.

This also involves checking the dependancies of the dependancies which are often overlooked.

Analyze

When acquiring or updating a library make sure to always check for yourself the status of the build.

If examining code directly is infeasible then check reviews, process, licenses and other aspects of the library project.

This early step can help prevent a lot of hassle compared to dealing with problems further down the line.

Control

A good process to have in place is a method by which only authorised libraries and versions can be used in projects.

A developer can request to have a library added to the list when a suitable level of analysis is complete.

The method to determine what ends up on this list should not be overly difficult as it will encourage users to bypass the restriction.

Monitor

Where analysis has shown it to be safe, all dependancies should be kept up to date.

Having to alter your code to take into account of an update is no excuse not to do so and is an accepted fact of using third party libraries.

As there are typically no automated methods to be informed of updates to libraries, it is the developer's job to keep up to date with the library's development status and news.

Tools

There are some automated tools that can keep you informed of the latest news and versions of dependancies.

While these are useful and can be a time-saver, they are no substitute for performing the checks yourself.

Tools

SBT can be used to keep dependancies up to date by using 'latest-integration' in place of the version number but this may not be wanted behaviour.

OWASP's Dependency Check is a tool that examines dependancies for any publically reported vulnerabilities for java libraries.

Sonatype's CLM is an automatic tool designed to manage dependancies throughout the development lifecycle.

"Bobby" is an in house tool created to perform a similar function as Dependancy Check for scala libraries.

Examples

What will follow are two famous examples of third party libraries introducing vulnerabilities.

There are typically no reports on more recent cases until the problem has been fixed to avoid more people attempting to exploit it.

Spring MVC (2013)

The vulnerability existed due to the behaviour of the Expression Language utilised by Spring.

Using an HTML request the intruder could directly inject code and execute it server side, potentially even taking control of the server.

This was a previously fixed issue that re-emerged due to the discovery of additional issues relating to the flaw.

At the time an estimated 1.3 million instances of the flawed component had been downloaded at the time.

Apache CXF (2012)

The vulnerability occurred whenever a user submitted a request without providing any identity token.

This allowed any web service to be executed with full permission without authentication.

This was also the second emergence of the issue with a prior occurance of a similar fault in 2010.

An estimated 4.2 million downloads of the framework occurred within a 12 month period in 2013.
